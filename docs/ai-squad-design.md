## **1. 사이트맵 (Sitemap)**

플랫폼은 크게 **[대시보드]**, **[워크스페이스(IDE)]**, **[프로젝트 관리]**의 3단계 구조로 나뉩니다.

* **메인 대시보드 (Project Hub)**
* 내 프로젝트 목록 (현재 상태 및 리스크 요약)
* 신규 프로젝트 생성 (프롬프트 입력형 기획 시작)


* **AI-Sync 워크스페이스 (Main IDE)**
* **비주얼 캔버스 (Visual Flow)**: 액션 노드 및 흐름도 설계 화면
* **에이전트 제어 센터 (Agent Relay)**: PM(Sisyphus)과의 채팅 및 워크플로우 모니터링
* **데이터 인사이트 판넬**: 정적/동적 데이터 관리 및 영향도 분석


* **히스토리 & 맥락 창고 (Context Archive)**
* 버전별 의사결정 로그 (배경, 사유, 대안 기록)
* 타임라인 복구 지점 관리

## **2. 페이지별 화면 구성 및 계층 구조**

핵심 화면인 **워크스페이스**를 중심으로 계층을 설계합니다.

### **[핵심 화면: 중앙 분할 비주얼 IDE]**

이 화면은 **'설계-실행-모니터링'**이 동시에 이루어지는 공간입니다.

| 영역 | 구성 요소 | 위치 및 계층 | 역할 |
| --- | --- | --- | --- |
| **GNB (Top)** | 프로젝트명, 버전 정보, 배포 버튼, 리스크 요약 지표 | 최상단 고정 | 현재 프로젝트의 전체 상태 확인 |
| **LNB (Left)** | 페이지 목록, 컴포넌트 라이브러리, 데이터 에셋 | 좌측 사이드바 | 프로젝트 구성 요소 네비게이션 |
| **Main Canvas (Center)** | **Interactive Action Flowchart** | 중앙 (가장 넓은 영역) | 사용자의 클릭, 스크롤 등 '액션'과 '기능'을 노드로 시각화 |
| **Chat Panel (Right)** | PM 에이전트(Sisyphus) 채팅 창 | 우측 가변 슬라이드 | AI에게 명령 전달 및 에이전트 간 업무 릴레이 브리핑 확인 |
| **Sub Panel (Bottom)** | 영향도 분석(Score), 데이터 스키마, 자연어 변경 요약 | 하단 탭 구조 | 선택한 노드에 대한 기술적 세부 정보 및 위험도 수치화 |

## **3. 에이전트 간 협업 과정의 시각화**

비개발자가 "AI가 지금 무엇을 하고 있는가?"를 실시간으로 파악할 수 있도록 **'릴레이 트래커'** 방식을 도입합니다.

* **진행 상태의 노드 하이라이트**: 중앙 캔버스에서 현재 작업 중인 기능 노드 주위에 에이전트 아이콘(기획, 설계, 개발, QA)이 위치하며 애니메이션 효과를 줍니다.
* **업무 전달(Hand-over) 애니메이션**: PM 에이전트가 "개발 완료, QA 단계로 진입합니다"라고 말하면, 캔버스 상에서 결과물(파일 아이콘)이 Coder 노드에서 QA 노드로 이동하는 시각적 연출을 사용합니다.
* **에이전트 상태 바**: 우측 채팅창 상단에 `[기획 완료] -> [설계 중] -> [코딩 대기] -> [QA 대기]` 형태의 스텝 바를 배치하여 전체 공정률을 표시합니다.

## **4. 사용자 유지보수 및 확장 관리 포인트 (Maintenance Points)**

비개발자가 안심하고 시스템을 확장할 수 있도록 3가지 지표를 관리 포인트로 제공합니다.

1. **리스크 스코어 (Risk Score Card)**:
* 기능 수정 시 영향 범위를 1~10점으로 표시합니다. (예: "로그인 로직 수정은 DB 결제 정보와 연결되어 리스크 8점입니다.")


2. **데이터 연결 지도 (Dependency Map)**:
* 특정 데이터를 클릭하면 해당 데이터가 사용되는 모든 기능 노드가 붉게 점등되어, 수정 시 '망가질 수 있는 지점'을 사전에 인지시킵니다.


3. **맥락 팝업 (Context Tooltip)**:
* 과거에 설계된 노드 위에 마우스를 올리면 `[수정 이력: 3회] [최종 결정 사유: 보안 강화 위해 API 방식 변경]` 등의 요약 정보가 나타납니다.

## **5. UX 동작 흐름 및 워크플로우**

신규 기능을 추가할 때의 표준 워크플로우(Happy Path)입니다.

### **[Step 1: 요구사항 입력]**

* **UX**: 사용자가 채팅창에 "게시물에 '좋아요' 기능을 넣고 싶어"라고 입력.
* **Flow**: PM(Sisyphus)이 요구사항을 분석하고 캔버스에 '좋아요 클릭' 액션 노드 초안을 생성.

### **[Step 2: 영향도 확인 및 승인]**

* **UX**: 캔버스에 점선으로 연결된 노드들이 나타나고, 하단 판넬에 "기존 DB 스키마에 컬럼 추가 필요, 리스크 3점" 메시지 출력.
* **Flow**: 사용자가 리스크 확인 후 '실행 승인' 버튼 클릭.

### **[Step 3: 에이전트 릴레이 가동]**

* **UX**: 캔버스의 노드가 깜빡이며 에이전트들이 순차적으로 작업을 진행. 각 단계 완료 시 채팅창에 자연어로 요약 보고.
* **Flow**: Architect(설계) -> Coder(코드 생성) -> QA(테스트 및 버그 수정).

### **[Step 4: 검증 및 최종 배포]**

* **UX**: QA 에이전트가 "테스트 완료" 보고를 하면, 사용자는 미리보기 화면에서 기능을 직접 테스트.
* **Flow**: 사용자의 '최종 배포' 클릭 시 현재의 논의 맥락(Decision Log)을 패키징하여 히스토리에 저장 후 서버 반영.
