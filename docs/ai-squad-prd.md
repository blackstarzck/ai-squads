## **[PRD] AI 기반 자가 유지보수형 개발 플랫폼: AI-Sync OpenDev**

### **1. 제품 개요 (Product Overview)**

본 프로젝트는 **'oh-my-opencode'**의 멀티 에이전트 협업 프레임워크를 활용하여, 코딩 지식이 없는 비개발자가 AI 에이전트 군단을 지휘해 직접 서비스를 개발하고 유지보수할 수 있는 **비주얼 IDE 플랫폼**입니다. 사용자는 단순히 코드를 생성하는 것을 넘어, 서비스의 전체 흐름과 데이터의 영향도를 시각적으로 파악하며 전문가의 도움 없이도 자신의 프로젝트를 안전하게 확장하고 관리할 수 있습니다.

---

### **2. 핵심 목표 (Goals)**

* **멀티 에이전트 오케스트레이션:** 역할이 분담된 전문 AI들이 PM(Sisyphus)의 지휘 아래 유기적으로 업무를 인수인계하며 프로젝트의 논리적 일관성을 유지합니다.
* **사용자 액션 중심의 시각적 설계:** 페이지 진입부터 클릭, 스크롤 등 모든 UX 흐름을 다이어그램화하여 비개발자가 시스템 아키텍처를 직관적으로 제어할 수 있게 합니다.
* **데이터 기반 리스크 관리:** 프로젝트 내 정적/동적 데이터의 중요도와 기능 간 연결 관계를 수치화하여 안전한 확장 및 수정 환경을 제공합니다.
* **맥락 기반의 버전 관리:** 변경 사항의 배경, 고려사항, AI와의 논의 맥락을 기록하여 장기적인 유지보수 자산으로 활용합니다.
* **1인 개발의 대중화:** 기획부터 유지보수까지 전 과정을 AI가 보조하여 비개발자도 1인 기업으로서 기술적 독립성을 확보하게 합니다.

---

### **3. 주요 문제점 및 해결 방안 (Problem & Solution)**

| 문제점 (Pain Point) | 해결 방안 (Solution) |
| --- | --- |
| **유지보수 및 확장성 취약:** AI로 코드를 짤 수는 있지만, 수정 시 어디가 망가질지 몰라 고도화가 어려움. | **영향도 분석 및 액션 플로우:** 수정 시 리스크 점수(1~10)를 산출하고, 사용자 액션 중심의 흐름도를 통해 수정 전후의 변화를 시각적으로 확인. |
| **단일 AI의 컨텍스트 한계:** 프로젝트 규모가 커지면 한 명의 AI가 모든 설계 맥락을 기억하지 못해 오류 발생. | **Sisyphus 기반 멀티 에이전트:** PM 에이전트가 업무 릴레이를 감시하고, 과거 결정 맥락을 참조하여 에이전트 간 논리적 충돌 방지. |
| **의사결정 배경 망각:** 시간이 지난 후 과거에 왜 특정 방식으로 기능을 구현했는지 알 수 없어 유지보수가 중단됨. | **맥락 중심 히스토리 기록:** 각 버전별 변경 배경, 배경 설명, 고려사항 등 '맥락 정보'를 패키징하여 보존. |

### **4. 핵심 기능 (Detailed Key Features)**

#### **① Sisyphus: PM급 오케스트레이터 및 업무 릴레이 시스템**

* **역할 분담 및 업무 할당:** 사용자의 요구사항을 분석하여 기획(Requirement), 설계(Architect), 개발(Coder), 검증(QA) 에이전트에게 업무를 배분합니다.
* **업무 릴레이(Workflow):** 각 에이전트는 담당 태스크 완료 시 결과물을 다음 단계 에이전트에게 자동 전달하며, PM은 이 과정을 실시간 모니터링합니다.
* **품질 게이트키핑:** 결과물이 목적과 어긋날 경우 PM이 즉각 개입하여 반려 및 수정을 명령하며, 사용자가 이해할 수 있는 언어로 진행 상황을 브리핑합니다.

#### **② 사용자 액션 기반 기능 흐름도 (Interactive Action Flowchart)**

* **액션 노드 시각화:** 페이지 진입(Landing), 클릭, 드래그, 스크롤 등 사용자의 액션에 의해 트리거되는 기능들을 노드와 화살표로 다이어그램화합니다.
* **동작 순서도(Sequence):** "버튼 클릭 → 데이터 검증 → DB 저장 → 알림"과 같은 내부 동작 프로세스를 시각적으로 표현하여 설계 구조를 한눈에 파악하게 합니다.
* **시각적 설계 인터페이스:** 사용자가 흐름도 위에서 직접 노드를 추가하거나 연결선을 수정하여 AI에게 새로운 로직을 지시할 수 있습니다.

#### **③ 영향도 기반 데이터 관리 엔진 (Impact Analysis)**

* **정적 데이터 관리:** DB 스키마, API 명세, 환경 설정 등 시스템의 뼈대가 되는 데이터를 리스트업하고 중요도를 부여합니다.
* **동적 데이터 추적:** 사용자 세션, 트래픽 로그, 실시간 데이터 흐름을 카테고리화하여 관리합니다.
* **리스크 스코어링:** 특정 액션 노드나 데이터를 수정할 때 연결된 기능들에 미칠 영향도를 점수화(1~10점)하여 경고 및 대안을 제시합니다.

#### **④ 맥락 중심 버전 관리 시스템 (Decision History Log)**

* **히스토리 아카이브:** 프로젝트 확장 및 유지보수 과정에서 생성된 모든 버전을 시계열로 기록합니다.
* **의사결정 패키징:** 각 변경점마다 **[변경 배경], [AI의 제안 및 고려사항], [최종 결정 사유]** 등 맥락 정보를 함께 저장합니다.
* **지능형 복구:** 과거 버전으로 복구 시, 당시의 논의 맥락을 요약 제공하여 사용자가 혼란 없이 이전 상태를 이해하고 작업할 수 있도록 돕습니다.

#### **⑤ 중앙 분할 비주얼 IDE 인터페이스 (Split-View Layout)**

* **상단/좌측 - 비주얼 워크플로우 캔버스:** 전체 페이지 구조와 사용자 액션 중심의 흐름도를 표시하며, 에이전트의 현재 작업 지점을 하이라이트합니다.
* **하단/우측 - 작업 및 데이터 워크스페이스:**
* **채팅 인터페이스:** PM 에이전트와의 실시간 소통창.
* **데이터 인사이트:** 선택된 노드와 연관된 정적/동적 데이터 리스트 및 영향도 수치 대시보드.
* **자연어 요약:** 코드 변경 사항을 비개발자가 이해하기 쉬운 문장으로 요약하여 출력.

### **5. 유지보수 및 확장 시나리오 (Operation Scenarios)**

1. **기능 확장 (Scale-up):**
* 사용자가 "상세 페이지 스크롤 시 추천 상품 표시"를 요청하면, PM이 흐름도 상에 '스크롤' 액션을 설계하고 데이터 영향도를 브리핑한 뒤 에이전트 릴레이를 시작합니다.


2. **맥락 참조 수정 (Contextual Maintenance):**
* 과거 보안상의 이유로 설계된 로직을 수정하려 할 때, 시스템이 기록된 '의사결정 맥락'을 근거로 리스크를 경고하고 대안을 제안합니다.


3. **자가 치유 (Self-Healing):**
* 장애 발생 시 QA 에이전트가 로그를 분석해 문제 노드를 흐름도 상에 표시하고, 코더 에이전트와 협업하여 복구 프로세스를 가동합니다.



---

### **6. 기대 효과**

* **비전문가의 기술 주도권:** 시각적 흐름도와 영향도 수치를 통해 전문가 없이도 시스템을 완벽히 제어하고 확장할 수 있습니다.
* **장기적 프로젝트 안정성:** 의사결정 맥락 기록을 통해 시간이 지나도 '왜 이렇게 만들어졌는지' 파악 가능한 투명한 관리가 이루어집니다.
* **1인 개발의 대중화:** 복잡한 인프라나 코딩 지식 없이 아이디어만으로 완성도 높은 서비스를 지속적으로 운영할 수 있습니다.
